## 3. 当前已确认的产品定位

`育英星宠` 不是通用积分 SaaS，而是育英学校定制的校园成长激励系统。

核心定位：

`学校荣誉体系 + 班级积分激励 + 萌宠成长展示`

产品形态明确为双端：

- `Admin`：Web 管理后台
- `Display`：运行在白板 / 触控一体机 / Windows 浏览器的大屏展示型 SPA

## 5. 开发目标

本期开发目标分为三层：

### 5.1 业务目标

- 支持学校进行班级激励与学生成长展示
- 支持班主任完成高频积分评价
- 支持学校管理层查看校级数据与成果
- 支持学校品牌展示与领导视察场景

### 5.2 产品目标

- 按原型完成双端核心闭环
- 保持学校红蓝主视觉和展示感
- 保证 Display 端可直接用于大屏展示
- 保证 Admin 端可支撑真实管理流程

### 5.3 技术目标

- 双端采用统一前端技术栈
- 后端采用标准 REST API 或 BFF 结构
- 数据模型支持学校、班级、学生、积分、荣誉、兑换、萌宠等核心域
- Display 端支持实时刷新或准实时同步

## 6. 总体架构建议

建议采用：

- 前端 1：`admin SPA`
- 前端 2：`display SPA`
- 后端：统一业务服务
- 数据库：关系型数据库
- 对象存储：图片、背景图、萌宠素材、奖励素材

推荐逻辑结构：

```text
浏览器 / 白板大屏
  ├─ Admin SPA
  └─ Display SPA
        ↓
     API Gateway / BFF
        ↓
     业务服务层
        ├─ 认证与权限
        ├─ 学校与班级管理
        ├─ 学生与评价
        ├─ 荣誉勋章
        ├─ 奖励兑换
        ├─ 萌宠图鉴
        ├─ 数据分析
        └─ 展示配置
        ↓
   MySQL / PostgreSQL + OSS / 本地对象存储
```

## 7. 技术选型定稿

以下技术选型已确认，后续开发文档与施工以此为准。

### 7.1 前端

定稿：

- 框架：`React`
- 构建工具：`Vite`
- 路由：`React Router`
- 状态管理：`Zustand`
- 请求层：`Axios` 或 `fetch` 封装
- Admin UI：`组件库 + 定制主题`
- Display UI：`完全自定义页面组件`

前端实施原则：

- `Admin` 追求管理效率、信息密度、稳定表单与表格能力
- `Display` 追求展示感、沉浸感、动效与大屏交互，不直接复用后台组件库
- 两端共享基础能力层，不强行共享视觉组件

建议目录：

```text
frontend/
  admin/
  display/
  shared/
```

### 7.2 后端

定稿：

- 语言：`Node.js`
- 框架建议：`NestJS`

原因：

- 模块化清晰，适合后台管理系统
- 鉴权、DTO、验证、模块拆分比较规范
- 便于后续扩展 WebSocket / 定时任务 / 文件上传

### 7.3 数据库

定稿：

- `MySQL 8+`

补充建议：

- ORM 可选 `Prisma` 或 `TypeORM`
- 若重视开发效率与类型安全，优先 `Prisma`

### 7.4 实时能力

定稿：

- 实时能力使用 `WebSocket`

建议用途：

- Display 端班级主页实时刷新
- Admin 端评价后同步更新 Display
- 排行榜、班级总积分、今日之星、目标进度实时变化
- 可选推送荣誉弹幕、兑换成功反馈、升级动画触发

## 9. 角色与权限模型

当前原型已明确以下角色：

- 超级管理员
- 学校管理员
- 德育负责人
- 年级负责人
- 班主任
- 任课教师
- 展示端账号

权限模型建议采用：

- `RBAC` 为主
- 必要时叠加班级/年级范围限制

权限控制分两层：

- 角色决定功能访问
- 数据范围决定能看哪些班级、学生、记录

### 9.1 Display 端权限原则

Display 端不能等同于后台。

原则：

- 只能登录到绑定班级
- 只允许有限展示型操作
- 不允许敏感后台配置
- 不允许跨班级管理

### 9.2 教师操作权限建议

当前原型中 Display 端支持：

- 单人加减分
- 批量加减分
- 按组加减分
- 分组管理
- 宠物领养
- 积分兑换

这些操作都不应仅停留在前端内存模拟，正式环境中必须有权限校验和操作日志。

- 规则应支持学期维度生效
- 规则删除建议采用软删除

规则来源补充：

- 第一阶段规则初始化必须参考 [`加扣分-教务模块（课堂、作业、学业）.xls`](/Users/wuwoo/Desktop/work/_育英星宠/YuYingPets/doc/加扣分-教务模块（课堂、作业、学业）.xls)
- 规则不是只分“加分 / 扣分”，而是要同时支持：
  - 通用教务规则
  - 学科组规则
  - 正向行为规则
  - 负向行为规则

当前从表格中已确认的规则层级：

- 通用教务模块：课堂、作业、学业
- 学科模块：
  - 语文组
  - 数学组
  - 英语组
  - 物理组
  - 化学组
  - 地理组
  - 生物组
  - 历史组
  - 政治组
  - 音美信综合组
  - 体育组

因此，积分规则表设计不能只支持“分类名称”，还需要支持：

- `module_type`：`general` / `subject`
- `subject_code`：如 `chinese` / `math` / `english`
- `scene_code`：如 `classroom` / `homework` / `exam` / `reading`
- `score_mode`：固定分值 / 区间分值 / 条件分值

基于表格抽取的通用教务规则示例：

- 课堂表现优秀：`+2`
- 作业优秀：`+2`
- 考试进步显著：`+5`
- 学科竞赛获奖：`+5 ~ +10`
- 早读认真：`+1`
- 上课迟到：`-2`
- 课堂违纪：`-2`
- 缺交/迟交作业：`-2`
- 考试作弊：`-10`
- 无故旷课：`-5`

基于表格抽取的学科规则特征：

- 存在大量学科专属正负向规则
- 存在“区间分值”，如 `+3~+10`、`扣1-4`
- 存在“阶段评价”，如周测、月评价、期末考核、听默写
- 存在“表现类行为”，如课堂状态、书写、背诵、展讲、答疑

这意味着正式系统不能把规则设计得过于死板。

## 15. 关键机制讨论与建议方案

本节用于收敛当前最关键的 3 个产品与技术问题：

- 教师如何在 `Display` 与 `Admin` 两端都进行加减分
- 宠物升级机制如何设计
- 如何基于加减分沉淀学生档案与 AI 学情评估

### 15.1 教师双端加减分与权限方案

核心建议：

- `Display` 与 `Admin` 都支持真实加减分
- 但两端能力边界不同
- 两端调用同一套评价接口
- 所有结果由后端统一计算，再通过 WebSocket 广播

#### 15.1.1 两端职责划分

##### Display 端

适合承载：

- 单人快捷加减分
- 批量加减分
- 按组加减分
- 分组管理
- 快速查看最近积分动态

不适合承载：

- 新建规则
- 大批量历史修订
- 跨班级查询
- 学期级/学校级分析
- 学生基础档案维护

##### Admin 端

适合承载：

- 单人或批量评价
- 历史记录查询与修正
- 学生档案查看
- 学情评估查看
- 规则维护
- 荣誉、奖励、图鉴、数据分析

#### 15.1.2 角色权限建议

- `班主任`
  - Display：可对本班执行快捷加减分、分组管理、兑换确认、萌宠领养
  - Admin：可对本班执行完整评价、查看档案、查看学情、管理本班学生
- `任课教师`
  - Display：可对授课班级执行快捷加减分，但不允许兑换、不允许修改班级展示配置
  - Admin：可对授课班级执行课堂评价，但不能改规则、不能改学生基础资料
- `年级负责人`
  - Admin：查看本年级评价情况、异常记录和学情汇总
- `德育负责人`
  - Admin：查看全校评价趋势、荣誉、奖励和学情汇总
- `学校管理员`
  - Admin：全局配置与管理
- `展示端账号`
  - Display：仅展示，不允许真实写操作

#### 15.1.3 推荐登录模式

我建议采用“双模式”：

- `展示模式`
  - 使用展示端账号或免登录进入
  - 只能查看，不能写
- `教师操作模式`
  - 由教师账号临时解锁
  - 可进行快捷加减分
  - 一段时间无操作后自动回退到展示模式

这比“Display 永久在线教师账号”更安全。

#### 15.1.4 统一评价接口

无论 Display 还是 Admin，都统一调用：

- `POST /api/score-records`
- `POST /api/score-records/batch`
- `POST /api/score-records/group`

每次评价记录必须包含：

- 学生 ID / 学生列表
- 班级 ID
- 规则 ID
- 分值
- 原因标签
- 来源端：`admin` / `display`
- 操作人 ID
- 时间

服务端统一处理流程：

1. 校验角色与数据范围权限
2. 写入积分记录
3. 更新学生成长汇总
4. 计算宠物成长
5. 判定是否触发荣誉
6. 更新班级排行
7. 广播 WebSocket 事件

#### 15.1.5 推荐结论

当前我建议定为：

- `Admin` 是完整评价端
- `Display` 是教师解锁后的快捷评价端，班主任和任课教师都可使用
- `展示端账号` 只展示，不写入

### 15.2 宠物升级机制方案

核心建议：

- `学生积分`
- `宠物等级 / 阶段`

采用“积分驱动成长阈值”的单轨模型，不再单独维护第二套成长口径。

#### 15.2.1 为什么建议积分直接驱动成长

- 萌宠成长口径与积分体系保持一致，配置和运营更简单
- 接口、规则、排行榜与成长阈值可以共用同一套数据
- AI 学情评估直接复用积分行为数据，不再额外维护第二套成长模型
- 通过“只升不降”可以避免扣分导致宠物回退的体验问题

#### 15.2.2 推荐成长模型

每次评价写一类变化：

- `scoreDelta`

示例：

- 课堂表现优秀：`+5 分`
- 助人为乐：`+3 分`
- 未做作业：`-3 分`

这样形成：

- 积分：排行、兑换、成长推进的统一口径

#### 15.2.3 升级规则建议

建议采用递增积分阈值：

- Lv1 -> Lv2：20 分
- Lv2 -> Lv3：50 分
- Lv3 -> Lv4：90 分
- Lv4 -> Lv5：140 分
- 后续逐级递增

同时为每个等级或阶段绑定素材：

- `pet_stage_1`
- `pet_stage_2`
- `pet_stage_3`
- ...

前端只展示服务端返回的：

- 当前等级
- 当前阶段图
- 距离下一级所需积分

#### 15.2.4 扣分与降级建议

第一期不建议做实时降级。

建议：

- 扣分影响当前积分
- 扣分进入学情评估
- 宠物等级默认不回退

如果后期想加负反馈，可增加：

- 活力值
- 状态值

而不是直接降级。

#### 15.2.5 第一阶段推荐结论

- `积分` 同时用于排行、兑换与成长
- `宠物等级` 只升不降

这是当前最稳的方案。

### 15.3 学生档案与 AI 学情评估方案

核心建议：

- 学情评估不能只看总分
- 要基于“行为轨迹 + 标签维度 + 时间序列 + 教师观察”

#### 15.3.1 学生档案建议拆成 4 层

##### 第一层：基础档案

- 姓名
- 班级
- 学号
- 年级
- 班主任

##### 第二层：成长档案

- 当前积分
- 累计积分
- 萌宠等级
- 获得勋章
- 奖励兑换记录

##### 第三层：行为档案

积分规则必须沉淀行为标签，例如：

- 课堂专注
- 作业质量
- 阅读习惯
- 协作能力
- 纪律表现
- 助人行为
- 表达能力
- 体育参与

也就是说，积分规则除了分值，还需要带：

- 行为维度
- 一级分类
- 二级标签
- 正负向属性

##### 第四层：AI 学情档案

AI 评估建议输出：

- 优势标签
- 风险标签
- 最近成长趋势
- 行为稳定度
- 建议关注点
- 家校沟通建议

#### 15.3.2 为 AI 预留的数据字段

积分规则表建议预留：

- `dimension`
- `tag`
- `sentiment`
- `weight`
- `ai_summary_text`

示例：

- 规则名：课堂表现优秀
- 分值：+5
- 维度：课堂学习
- 标签：专注投入
- 正负向：正向
- 权重：0.8
- AI 解释：该学生在课堂参与和专注度方面表现较好

#### 15.3.3 AI 数据来源建议

AI 评估输入建议综合：

- 近 7 天 / 30 天积分记录
- 正向/负向行为分布
- 行为维度统计
- 成长趋势
- 勋章与荣誉记录
- 奖励兑换记录
- 教师补充评语

建议在 Admin 学生详情页预留：

- 教师观察记录
- 阶段性评语
- 特殊关注事项

#### 15.3.4 AI 输出形式建议

第一期不建议做“强判定式”学生标签，而建议先做：

- 周报摘要
- 月度成长总结
- 风险提醒
- 教师建议草稿

例如：

- 本周该生在课堂参与和作业完成方面持续稳定，合作类行为明显提升。
- 近两周负向记录主要集中在课堂纪律，建议重点关注下午时段状态波动。
- 可考虑给予一次公开表扬，强化其在阅读和表达方面的积极反馈。

#### 15.3.5 AI 技术路线建议

第一期推荐采用“规则统计 + AI 总结”的混合方案：

1. 后端先做结构化统计
2. 生成稳定的学生行为摘要 JSON
3. 再交给 AI 输出教师可读结论

不建议一开始直接把原始积分流水喂给 AI。

#### 15.3.6 推荐结论

第一期建议把学生档案定义为：

- `成长档案`
- `行为档案`
- `AI 学情档案`

这会让系统从“积分工具”升级为“成长管理系统”。

当前阶段已确认：

- AI 学情评估第一阶段只做“摘要与建议”
- 不直接输出强判定式等级或风险分层
- 输出对象优先为教师与管理者

## 16. 正式环境与原型的差异处理

必须明确，当前 `prototype` 是高质量静态原型，不是业务逻辑实现。

原型中的以下逻辑正式开发时必须替换：

### 16.1 前端内存数据

当前原型中的：

- 学生数组
- 萌宠选择
- 积分变更
- 排行变化
- 兑换扣分

全部是前端内存数据模拟。

正式环境中必须改为：

- 页面初始化从接口加载
- 所有变更通过接口提交
- 成功后刷新局部状态

### 16.2 模拟成功提示

例如兑换成功文案中的“仅在展示端模拟反馈体验”，正式版必须去掉。

### 16.3 前端直接决定升级

当前原型里学生积分达到阈值会直接前端升级。

正式环境建议：

- 升级规则由后端统一判定
- 前端只展示结果

这样可以避免 Display 端和 Admin 端规则不一致。

## 17. 非功能要求

### 17.1 视觉一致性

- 保持育英学校红蓝主色
- Admin 偏正式、展示型
- Display 偏沉浸、舞台化

### 17.2 性能

- Display 首屏要快
- 动画不能卡顿
- 班级主页数据刷新要稳定

### 17.3 兼容性

- Admin：现代浏览器
- Display：1920 宽度优先，大屏横屏优先

### 17.4 安全

- 所有写操作需鉴权
- 核心操作留痕
- 密码加密存储

### 17.5 日志

至少记录：

- 登录
- 规则变更
- 加减分
- 领养萌宠
- 奖励兑换
- 配置变更

## 18. 实施顺序建议

为了降低风险，建议按下面顺序开发。

### 阶段 1：后端基础能力

### 阶段 4：荣誉与奖励

## 20. 进入施工前待确认项

这是目前最重要的一节。以下问题不建议带着模糊状态直接开工。

### 20.1 Display 登录账号

要确认：

- 使用班主任后台账号直接登录
- 还是为每个班单独配置展示端账号

我更建议：

- 教师后台账号和展示端账号分离

原因：

- 权限更安全
- 白板上登录风险更低

### 20.2 Display 写操作范围

现已确认：

- Display 允许真实加减分
- 班主任和任课教师都可以在 Display 端执行加减分
- 兑换只允许班主任账号执行
- 萌宠领养按班主任权限执行

因此，Display 已经不是纯展示端，而是“展示 + 快捷教学操作端”，权限、日志、自动锁定策略必须提高一级。

### 20.3 兑换是否需要审核

现已确认：

- 第一阶段兑换即时生效
- 不增加后台审核流
- 仅允许班主任账号在 Display 端执行兑换
- Admin 端负责记录查询与事后追踪

### 20.4 积分与萌宠等级关系

要确认：

- 萌宠等级是否直接由累计积分阈值驱动

我建议：

- 学生积分与萌宠等级保持关联，但采用“累计积分达阈值升级”
- 萌宠成长阈值由系统设置统一配置，不再维护独立成长值模型

这样可避免后续玩法过于死板。

### 20.5 学生是否允许“待领养”

原型中存在未领养学生。

要确认：

- 所有学生初始化时是否必须绑定萌宠
- 还是允许后续领养

### 20.6 汇报展示模式是否单独开发

原型中 Admin 驾驶舱带“汇报展示模式”按钮。

要确认：

- 只是同页全屏
- 还是一套独立汇报视图

如果是后者，需要单独定义页面结构和切换策略。

### 20.7 动画是否必须完全还原原型

要确认：

- 过场动画是否必须 1:1 还原
- 还是先实现简化版

如果本期优先业务落地，建议先做可配置简化版。

## 21. 结论

基于目前的原型和既有文档，`育英星宠` 已经具备进入开发评审的条件，但还不适合直接开工。

原因不是方向不清，而是有几项会直接影响架构和权限边界的关键决策，还需要先拍板：

- Display 是纯展示端，还是轻业务端
- 兑换是否审核
- 登录账号体系如何设计
- 萌宠成长模型如何定义
- 汇报展示模式是否独立建模

这些问题一旦确认，这份开发文档就可以继续细化为：

- 数据库表结构文档
- 接口详细文档
- 前端项目结构文档
- 实施排期文档

在这些确认前，不建议直接施工。
