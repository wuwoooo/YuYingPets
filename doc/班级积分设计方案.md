# 班级积分设计方案

## 1. 背景与目标

当前系统中的“班级积分排行”主要由班级内学生个人当前积分求和得到：

- `class.currentScoreTotal` 来源于 `student_profile.current_score` 聚合。
- 个人加扣分记录写入 `score_record`，并强制绑定 `student_id`。
- 展示端排行榜 `/display/classes/:classId/leaderboard` 是班内学生榜，不是班级榜。

本次新增“班级积分”后，需要把真正的班级评价从个人积分体系中拆出，形成独立闭环。术语上需要同步调整：当前 admin-web 中面向班内学生、分组、批量学生的“班级评价”应改名为“学生评价”；只有对班级积分进行评分的功能才叫“班级评价”。

- 班级积分是班级维度的独立评价分，不影响学生个人积分、宠物成长、个人兑换。
- 评分体系中部分规则可被标识为“班级积分评价”。
- 学校可基于这些规则直接给班级加分或扣分。
- 班级排名改为依据班级积分，而不是班级内学生个人积分总和。
- 班级排名按年级内排名，不做全校混排。

## 2. 核心概念

### 2.1 评价对象

建议为评分规则增加评价对象字段，而不是只增加一个布尔值。

```text
score_target = student | class
```

- `student`：个人积分规则，沿用现有学生评价链路；前端统一命名为“学生评价”。
- `class`：班级积分规则，只能作用于班级。

这样比 `is_class_score_rule` 更稳，因为未来如果增加“小组积分”“宿舍积分”，可以扩展为新的 target，而不是继续堆布尔字段。

### 2.2 班级积分账户

每个班级在每个学期拥有一套独立积分账户：

- `current_score`：当前班级积分，可被加扣分影响，允许为负分。
- `total_score`：累计正向班级积分，只累计加分，用于长期荣誉、阶段成就等。
- `positive_count_7d` / `negative_count_7d`：近 7 天正负向评价次数，供分析看板使用。
- `last_score_at`：最近一次班级积分变动时间。

## 3. 数据模型设计

### 3.1 扩展 `score_rule`

新增字段：

```sql
ALTER TABLE score_rule
  ADD COLUMN score_target ENUM('student', 'class') NOT NULL DEFAULT 'student' AFTER score_mode,
  ADD INDEX score_rule_score_target_idx(score_target);
```

Prisma 建议新增：

```prisma
enum ScoreTarget {
  student
  class
}

model ScoreRule {
  scoreTarget ScoreTarget @default(student) @map("score_target")
}
```

规则约束：

- `score_target = student` 时，走现有个人加扣分流程，对应前端“学生评价”。
- `score_target = class` 时，走班级加扣分流程，对应前端“班级评价”。
- `module_type`、`subject_code`、`scene_code` 可继续复用，班级评价仍可归入纪律、卫生、活动、竞赛等场景。
- 后台创建规则时必须明确选择“个人积分”或“班级积分”。

### 3.2 新增 `class_score_profile`

用于保存班级积分账户的当前状态。

```sql
CREATE TABLE class_score_profile (
  id BIGINT PRIMARY KEY AUTO_INCREMENT,
  school_id BIGINT NOT NULL,
  semester_id BIGINT NOT NULL,
  class_id BIGINT NOT NULL,
  current_score INT NOT NULL DEFAULT 0,
  total_score INT NOT NULL DEFAULT 0,
  positive_count_7d INT NOT NULL DEFAULT 0,
  negative_count_7d INT NOT NULL DEFAULT 0,
  last_score_at DATETIME(3) NULL,
  updated_at DATETIME(3) NOT NULL,
  UNIQUE KEY class_score_profile_class_id_key(class_id),
  INDEX class_score_profile_school_semester_grade_idx(school_id, semester_id),
  CONSTRAINT class_score_profile_class_id_fkey FOREIGN KEY (class_id) REFERENCES class(id)
);
```

Prisma 建议：

```prisma
model ClassScoreProfile {
  id              BigInt    @id @default(autoincrement())
  schoolId        BigInt    @map("school_id")
  semesterId      BigInt    @map("semester_id")
  classId         BigInt    @unique @map("class_id")
  currentScore    Int       @default(0) @map("current_score")
  totalScore      Int       @default(0) @map("total_score")
  positiveCount7d Int       @default(0) @map("positive_count_7d")
  negativeCount7d Int       @default(0) @map("negative_count_7d")
  lastScoreAt     DateTime? @map("last_score_at")
  updatedAt       DateTime  @updatedAt @map("updated_at")

  school    School    @relation(fields: [schoolId], references: [id])
  semester  Semester  @relation(fields: [semesterId], references: [id])
  classroom Classroom @relation(fields: [classId], references: [id])

  @@index([schoolId, semesterId])
  @@map("class_score_profile")
}
```

说明：

- `class_id` 当前已经属于具体学期，因此 `class_id` 唯一即可。
- 如果未来允许同一个物理班级跨学期复用同一个 `class_id`，再改为 `@@unique([semesterId, classId])`。

### 3.3 新增 `class_score_record`

不要把班级积分记录塞进现有 `score_record`。现有表强制绑定学生，并会触发学生个人积分、宠物成长、个人排行榜等副作用。班级积分应有独立流水表。

```sql
CREATE TABLE class_score_record (
  id BIGINT PRIMARY KEY AUTO_INCREMENT,
  school_id BIGINT NOT NULL,
  semester_id BIGINT NOT NULL,
  class_id BIGINT NOT NULL,
  rule_id BIGINT NOT NULL,
  subject_code VARCHAR(32) NULL,
  scene_code VARCHAR(32) NULL,
  dimension VARCHAR(64) NULL,
  tag VARCHAR(64) NULL,
  sentiment ENUM('positive', 'negative') NOT NULL,
  score_delta INT NOT NULL,
  remark VARCHAR(255) NULL,
  source_terminal ENUM('admin', 'display') NOT NULL,
  source_role VARCHAR(32) NULL,
  operator_id BIGINT NOT NULL,
  operator_name VARCHAR(64) NULL,
  created_at DATETIME(3) NOT NULL DEFAULT CURRENT_TIMESTAMP(3),
  INDEX class_score_record_class_created_idx(class_id, created_at),
  INDEX class_score_record_rule_id_idx(rule_id),
  INDEX class_score_record_school_semester_idx(school_id, semester_id),
  CONSTRAINT class_score_record_class_id_fkey FOREIGN KEY (class_id) REFERENCES class(id),
  CONSTRAINT class_score_record_rule_id_fkey FOREIGN KEY (rule_id) REFERENCES score_rule(id),
  CONSTRAINT class_score_record_operator_id_fkey FOREIGN KEY (operator_id) REFERENCES user(id)
);
```

### 3.4 班级积分批量记录

班级积分需要支持批量给多个班级同时加扣分，建议新增批量主表：

```text
class_score_record_batch
class_score_record_batch_item
```

批量操作必须保留审计分组，便于后台查看“一次检查/评比”影响了哪些班级。

## 4. 业务流程设计

### 4.1 评分规则维护

后台“评分体系”增加字段：

- 评价对象：`个人积分` / `班级积分`
- 当选择 `班级积分`：
  - 不显示学生、宠物成长相关说明。
  - 可归类到“纪律、卫生、活动、竞赛、班级建设”等场景。
  - 是否展示到班级大屏由 `display_enabled` 继续控制。

规则列表支持筛选：

- 全部
- 个人积分
- 班级积分

### 4.2 班级加扣分

新增班级评价入口，建议放在：

- 后台 Dashboard 或 Evaluation 页面增加“班级评价”模式。
- 班级列表页的每个班级行增加“评价”操作。

注意：admin-web 当前面向学生的“班级评价”入口需要更名为“学生评价”。本节所说“班级评价”专指对 `class_score_profile` 加扣分的班级积分评价。

提交参数：

```json
{
  "classId": 101,
  "ruleId": 22,
  "remark": "本周卫生检查优秀",
  "sourceTerminal": "admin"
}
```

后端处理：

1. 校验登录用户有权限访问目标班级。
2. 校验规则存在且 `score_target = class`。
3. 校验用户角色可使用该规则。
4. 根据 `score_type` 和 `score_value` 计算有符号分值。
5. 写入 `class_score_record`。
6. upsert `class_score_profile`：
   - `current_score += score_delta`
   - `total_score += max(score_delta, 0)`
   - 正负向次数累加
   - 更新 `last_score_at`
7. 写入操作日志。
8. 发送实时事件：
   - `grade.class_score.changed`
   - `class.score.changed`

班级积分允许扣为负数，后端不得把 `current_score` 截断为 0。

### 4.3 班级批量加扣分

批量提交参数：

```json
{
  "classIds": [101, 102, 103],
  "ruleId": 22,
  "remark": "本周卫生检查优秀",
  "sourceTerminal": "admin"
}
```

后端处理：

1. 校验操作者拥有全部目标班级的班级积分操作权限。
2. 校验规则存在且 `score_target = class`。
3. 创建 `class_score_record_batch`。
4. 为每个班级创建一条 `class_score_record`。
5. 为每条记录创建 `class_score_record_batch_item`。
6. 分别 upsert 每个班级的 `class_score_profile`。
7. 写入操作日志，日志目标为批量记录。
8. 按涉及年级发送排行榜变更事件。

### 4.4 个人评价防串线

现有 `/score-records` 个人评价接口必须增加保护：

- 如果传入规则 `score_target = class`，返回错误：`班级积分规则不能用于学生评价`。

新增 `/class-score-records` 接口也必须增加保护：

- 如果传入规则 `score_target = student`，返回错误：`个人积分规则不能用于班级评价`。

这是本设计最关键的边界，避免班级积分污染个人积分。

## 5. 排名设计

### 5.1 班级排名口径

班级排名使用：

```text
class_score_profile.current_score
```

不再使用：

```text
sum(student_profile.current_score)
```

排序规则：

1. `current_score` 降序。
2. 同分班级名次相同，例如两个班级同为最高分时都显示第 1 名。
3. 同分班级内部展示顺序按 `class.sort_order` 升序、`class.id` 升序稳定排列，但不影响名次编号。

### 5.2 年级内排名

接口必须带年级上下文，或从当前班级推导年级：

```http
GET /class-score-records/rankings?gradeCode=G7
GET /classes/:classId/class-score-ranking
```

返回示例：

```json
{
  "gradeCode": "G7",
  "gradeName": "七年级",
  "rows": [
    {
      "rank": 1,
      "classId": 101,
      "className": "七年级一班",
      "currentScore": 86,
      "totalScore": 124,
      "lastScoreAt": "2026-04-30T08:20:00.000Z"
    }
  ]
}
```

展示端如果在某个班级大屏内打开“班级榜”，后端根据该班级的 `grade_code` 查询同年级班级，不返回全校班级。

## 6. 接口设计

### 6.1 评分规则

扩展现有接口：

```http
GET /score-rules?scoreTarget=class
POST /score-rules
PATCH /score-rules/:id
```

请求/响应增加：

```json
{
  "scoreTarget": "class"
}
```

### 6.2 班级积分流水

新增：

```http
GET /class-score-records?classId=101
POST /class-score-records
POST /class-score-records/batch
POST /class-score-records/:id/reverse
```

本次完整交付必须实现：

- `GET /class-score-records`
- `POST /class-score-records`
- `POST /class-score-records/batch`

`reverse` 当前个人积分也只是占位，建议等个人和班级撤销统一设计后一起做。

### 6.3 班级积分排名

新增：

```http
GET /class-score-records/rankings?gradeCode=G7
GET /display/classes/:classId/class-score-ranking
```

后台可直接按 `gradeCode` 查询；展示端按 `classId` 推导年级，避免前端传错年级导致越权或混榜。

### 6.4 班级列表

`GET /classes` 当前返回：

```json
{
  "currentScoreTotal": 230
}
```

建议改为兼容返回：

```json
{
  "classScore": 86,
  "classTotalScore": 124,
  "studentCurrentScoreTotal": 230,
  "studentTotalScoreTotal": 520
}
```

兼容期可以保留 `currentScoreTotal`，但语义要明确：

- 方案 A：`currentScoreTotal` 暂时继续代表学生积分总和，新增 `classScore` 给班级排名使用。
- 方案 B：`currentScoreTotal` 改为班级积分，另加 `studentCurrentScoreTotal`。

推荐方案 A，避免破坏现有页面；所有“班级排行”页面显式改用 `classScore`。

## 7. 前端页面调整

### 7.1 评分体系页

需要新增：

- 评价对象字段。
- 列表标签：个人积分 / 班级积分。
- 筛选条件：评价对象。
- 表单联动：班级积分规则隐藏学生成长相关提示。

### 7.2 评价页

admin-web 当前面向学生的“班级评价”需要更名为“学生评价”。调整后评价页建议保留两个清晰入口：

```text
学生评价 | 班级评价
```

学生评价：

- 承接当前 admin-web 已有的“班级评价”功能。
- 使用现有学生、分组、批量学生操作。
- 只加载 `scoreTarget=student` 的规则。
- 产生 `score_record`，影响学生个人积分、宠物成长、学生排行榜。

班级评价：

- 左侧选择年级。
- 中间显示年级内班级列表。
- 右侧选择班级积分规则。
- 支持单班评价和多班批量评价。
- 提交后刷新班级积分、年级班级排行和班级积分流水。
- 只加载 `scoreTarget=class` 的规则。
- 产生 `class_score_record`，只影响班级积分，不影响任何学生个人积分。

### 7.3 Dashboard / 数据分析

当前多个位置使用 `currentScoreTotal` 做“明星班级”和“班级积分排行”。需要改为：

- 明星班级：按 `classScore` 排序。
- 年级对比：使用各年级班级积分均值，避免班级数量多的年级天然更高。
- 学生榜：继续按个人积分。
- 班级健康分析：可以同时展示“班级积分”和“学生积分总和”，但不能混为一个排名口径。

### 7.4 展示端

现有展示端的“光荣排行榜”是班内学生榜，建议保留。

新增“年级班级榜”入口：

- 当前班级所在年级内展示所有班级排名。
- 当前班级高亮。
- 分数显示为班级积分。

展示端榜单标题建议：

- `班级积分榜 · 七年级`
- 避免叫“全校班级榜”。

## 8. 权限设计

建议规则：

- `super_admin` / `school_admin` / `moral_admin`：可给全校任意班级做班级评价。
- `grade_admin`：可给负责年级内班级做班级评价。
- `homeroom_teacher`：不能操作班级积分。
- `subject_teacher`：不能操作班级积分。
- `display_account`：不能执行加扣分。

权限实现可复用现有 `ensureCanAccessClass` 和 `ensureCanUseRuleForClass`，但班级积分写入必须额外限制角色白名单：

```text
super_admin | school_admin | moral_admin | grade_admin
```

其中 `grade_admin` 仍需要受年级范围约束。班主任和任课老师只能查看其权限范围内的班级积分与排名，不能创建、批量创建或撤销班级积分记录。

## 9. 实时事件设计

新增事件：

```text
class.score.changed
grade.class_ranking.changed
```

事件载荷：

```json
{
  "classId": 101,
  "gradeCode": "G7",
  "scoreDelta": 5,
  "currentScore": 86,
  "operatorName": "张老师"
}
```

展示端订阅当前班级房间时，可以收到当前班级积分变动；如果打开年级班级榜，再触发榜单刷新。

## 10. 迁移策略

### 10.1 数据库迁移

1. 为 `score_rule` 添加 `score_target`，默认 `student`。
2. 新建 `class_score_profile`。
3. 新建 `class_score_record`。
4. 为已有班级初始化 `class_score_profile`，初始值为 0。

默认不把历史学生积分总和迁移为班级积分，因为新规则要求“独立于个人积分体系”。所有已有班级的班级积分统一从 0 开始。

### 10.2 前端兼容

本功能不拆阶段实施，但可以在一次完整交付内保留兼容字段，降低现有页面改造风险：

- 保留现有 `currentScoreTotal` 字段，继续代表学生个人积分总和。
- 新增 `classScore` / `classTotalScore`，代表班级积分。
- 所有“班级排行 / 明星班级 / 年级班级榜”必须在本次交付内统一改用 `classScore`。
- 当前 admin-web 中面向学生的“班级评价”文案必须在本次交付内统一改为“学生评价”。

## 11. 单阶段实施范围

本功能不分三阶段，以下内容作为一个阶段一次性交付完成。

### 11.1 数据与后端闭环

- `score_rule.score_target`
- `class_score_profile`
- `class_score_record`
- `class_score_record_batch`
- `class_score_record_batch_item`
- `POST /class-score-records`
- `GET /class-score-records`
- `GET /class-score-records/rankings`
- `POST /class-score-records/batch`
- 个人评价接口禁止使用班级积分规则

### 11.2 后台管理

- 评分体系支持评价对象。
- 当前面向学生的“班级评价”统一更名为“学生评价”。
- 新增真正的“班级评价”，用于班级积分加扣分。
- 班级评价支持单班和批量操作。
- 班级列表返回并展示班级积分。
- Dashboard 班级榜改用班级积分。

### 11.3 展示端与分析

- 展示端新增年级班级榜。
- 数据分析页班级排名改用班级积分。
- 年级对比使用班级积分均值。
- 实时刷新年级班级榜。

## 12. 已确认产品口径

1. 班级积分允许负分。
2. 年级对比看板使用班级积分均值。
3. 班级积分只能由校级管理员、德育管理员、年级管理员等管理角色操作；班主任和任课老师不能操作。
4. 班级积分需要支持批量加扣分。
5. 上线时已有班级积分全部从 0 开始。
6. 班级积分相同时排名相同，同分行只做稳定展示排序。
