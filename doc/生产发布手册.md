# 生产发布手册

适用项目：`YuYingPets`  
当前生产服务器：`8.137.161.101`

## 1. 原则

日常发布默认只走仓库根目录的 [`deploy.sh`](/Users/wuwoo/Desktop/work/_育英星宠/YuYingPets/deploy.sh)。

不要再把下面这些动作当成常规流程：

- 手工 `rsync` 覆盖 backend 目录
- 手工 `pm2 start dist/main.js`
- 手工补跑一串临时命令后不回收进脚本

## 2. 当前命令入口

在项目根目录执行：

```bash
cd /Users/wuwoo/Desktop/work/_育英星宠/YuYingPets
```

### 管理端

```bash
./deploy.sh admin-web
```

用途：

- 本地构建 `admin-web`
- 上传静态文件
- 校验登录页

### 大屏端

```bash
./deploy.sh display-app
```

用途：

- 本地构建 `display-app/web`
- 更新静态资源版本参数
- 上传静态文件
- 校验 `/display/`

### 后端默认发布命令

```bash
./deploy.sh backend-release
```

这是后端的默认入口。它会顺序执行：

1. `backend-precheck`
2. `backend-assets`
3. `backend-migrate`
4. `backend`

其中：

- 没有资产变化会自动跳过 `backend-assets`
- 没有 Prisma 增量迁移会自动跳过
- 已登记过的 `sql/migrate_*.sql` 会自动跳过

### 只同步后端静态资产

```bash
./deploy.sh backend-assets
```

用途：

- 只同步 `backend/public/assets/`
- 不重启后端
- 不执行数据库迁移

### 只执行数据库迁移

```bash
./deploy.sh backend-migrate
```

用途：

- 执行 `prisma migrate deploy`
- 执行并登记 `backend/sql/migrate_*.sql`

### 只发布后端代码

```bash
./deploy.sh backend
```

用途：

- 构建后端
- 上传到新 release
- 切换 `/www/wwwroot/yuyingpets/backend` 符号链接
- 重启 pm2
- 校验健康检查

### 只做首次迁移预检

```bash
./deploy.sh backend-precheck
```

用途：

- 查看线上 `backend` 是目录还是符号链接
- 查看 `shared/backend` 是否存在
- 查看资产、上传、日志目录状态

## 3. 当前后端目录结构

当前后端已经切换到 release 模式：

- 运行入口：`/www/wwwroot/yuyingpets/backend`
- 当前版本：符号链接指向 `releases/backend/<timestamp>`
- 共享目录：`/www/wwwroot/yuyingpets/shared/backend`
- 迁移工作目录：`/www/wwwroot/yuyingpets/shared/backend-migrate`

共享目录承载：

- `.env`
- `logs`
- `public/assets`
- `public/uploads`

这意味着：

- 后端代码版本切换不会覆盖上传目录
- 后端代码版本切换不会覆盖共享资产

## 4. SQL 迁移规则

### Prisma 迁移

自动执行：

```bash
npx prisma migrate deploy
```

### 手写 SQL

只自动处理：

- `backend/sql/` 下匹配 `migrate_*.sql` 的文件

执行策略：

- 使用 `deploy_sql_history` 登记表记录是否已执行
- 已执行文件会跳过
- 如果同名 SQL 已登记，但内容发生变化，脚本会报错并停止

### 不会自动执行的 SQL

以下默认不在 `backend-release` 自动流程里：

- `seed_*.sql`
- 导入类 SQL
- 一次性修复脚本
- 高风险业务数据重置脚本

这类操作需要单独评估后执行。

## 5. 常见发布顺序

### 只发后端代码改动

```bash
./deploy.sh backend-release
```

### 发后端代码 + 新资产

还是直接：

```bash
./deploy.sh backend-release
```

如果 `backend/public/assets/` 有变化，脚本会自动同步。

### 只补跑迁移

```bash
./deploy.sh backend-migrate
```

### 全站发布

```bash
./deploy.sh admin-web
./deploy.sh display-app
./deploy.sh backend-release
```

不要默认用 `all` 当作第一次排障命令。  
排障时更适合按模块单独执行。

## 6. 出问题时先看什么

### 后端发布失败

优先看：

1. `backend-migrate` 是不是失败
2. `backend-assets` 是不是失败
3. `backend` 切换后健康检查是否通过

### 线上健康检查

```bash
curl -k https://www.dlbfyy.cn/api/v1/health
```

### PM2 状态

```bash
pm2 status yuyingpets-backend
```

### 当前 backend 指向哪个版本

```bash
readlink -f /www/wwwroot/yuyingpets/backend
```

### 查看共享目录

```bash
ls -la /www/wwwroot/yuyingpets/shared/backend
```

## 7. SSH 现状

当前目标状态：

- `root` 走公钥登录
- `PasswordAuthentication no`
- `PermitRootLogin prohibit-password`

发布时默认使用：

```bash
~/.ssh/id_ed25519_tencent
```

## 8. 当前约束

1. `backend-release` 是后端默认发布命令
2. 不要手工覆盖 `/www/wwwroot/yuyingpets/backend`
3. 不要把 `public/uploads` 当作代码资源目录发布
4. 新增长期运维动作，优先回收进 `deploy.sh`
